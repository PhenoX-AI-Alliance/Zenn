2026年7月現在、AI駆動開発（AI-Driven Development）は単なるコード生成の域を超え、**「設計判断の自動化」**というフェーズに突入しました。本記事では、Anthropicの「Claude Code」を駆使し、DDD（ドメイン駆動設計）の集約設計において「地球環境レジリエンス」を評価指標として組み込む手法を解説します。

---

## 1. 2026年7月版：Claude CodeによるAI駆動開発の現在地

Claude Codeは、単なるチャットボットから「自律的なエージェント」へと進化しました。現在の開発現場では、以下の3点が標準となっています。

*   **コンテキスト認識の深化:** プロジェクトのアーキテクチャ全体（`src/domain`配下など）を深く理解し、制約事項を遵守したリファクタリングを提案。
*   **自律的なテスト駆動:** 修正案に対して即座にテストコードを生成し、実行結果に基づいて自己修正を行うループの高速化。
*   **非機能要件のコード化:** セキュリティやパフォーマンスだけでなく、**環境負荷（カーボンフットプリント）**といったビジネス要件外の指標を、CI/CDパイプラインの一部としてコードに埋め込むことが可能になりました。

---

## 2. DDDに「地球環境レジリエンス」を組み込む

従来のDDDでは「ビジネスロジックの整合性」が唯一の正義でした。しかし、これからのシステム設計では、**「そのドメインモデルは地球環境に対して持続可能か？」**という問いが不可欠です。

私たちは、集約（Aggregate）設計のバリデーションプロセスに「推定エネルギー消費量」を組み込むアプローチを採用しました。

### 設計判断のフロー
1.  **集約の変更:** Claude Codeに対し、ドメインモデルの変更を提案させる。
2.  **環境負荷計算:** 変更が引き起こすDBクエリ回数やデータ転送量をベースに、CO2排出量を概算。
3.  **ガードレール:** 設定された閾値を超えた場合、CI上でビルドエラーを発生させる。

---

## 3. 実装例：環境負荷を考慮した集約バリデーション

以下は、集約 `Order` を更新する際、処理に伴う計算リソースを評価し、環境負荷が高い場合に警告を出すバリデーションロジックの例です。

```typescript
// domain/order/OrderAggregate.ts

export class Order {
  // ...既存のビジネスロジック

  /**
   * 環境負荷を考慮したバリデーション
   * 処理の複雑度からCO2排出量（g）を概算
   */
  public validateEnvironmentalImpact(operationType: 'CREATE' | 'UPDATE'): void {
    const estimatedEnergyKWh = this.calculateEstimatedEnergy(operationType);
    const co2EmissionGrams = estimatedEnergyKWh * 450; // 日本の平均排出係数想定

    const THRESHOLD_G = 0.5; // 1操作あたりの許容排出量

    if (co2EmissionGrams > THRESHOLD_G) {
      throw new EnvironmentalImpactViolationError(
        `環境負荷が許容範囲を超えています: ${co2EmissionGrams.toFixed(2)}g CO2. 
         集約の設計を見直し、クエリの最適化を行ってください。`
      );
    }
  }

  private calculateEstimatedEnergy(op: string): number {
    // 複雑度に基づくエネルギー消費計算ロジック
    // 実際にはClaude Codeがプロジェクトのクエリ統計から係数を算出
    return op === 'UPDATE' ? 0.6 : 0.2; 
  }
}
```

Claude Codeはこのロジックに対し、`"集約の粒度が大きすぎてクエリ回数が増加しています。リポジトリ層でキャッシング戦略を導入してください"`といった具体的な設計改善案を自律的に提示してくれます。
