地球環境データ（気温、湿度、CO2濃度など）は、センサーのメーカーや設置場所によってフォーマットがバラバラです。これらをそのままデータベースに投入すると、後の分析や予測モデルの構築で大きな負債となります。

本記事では、Domain-Driven Design (DDD) の概念を用いて、センサーデータを適切に正規化し、ドメイン層でバリデーションを行うアーキテクチャを紹介します。

---

## なぜ環境データにDDDが必要なのか？

環境データには「単位の不一致」「異常値の混入」「精度の違い」という課題があります。単なるデータ構造（DTO）として扱うのではなく、**ドメインオブジェクト**として扱うことで、以下のメリットが得られます。

1. **整合性の保証**: 負の湿度は存在しない、といったルールをオブジェクト生成時に強制できる。
2. **計算の局所化**: 摂氏から華氏への変換や、特定の基準値への正規化ロジックをドメイン内に閉じ込める。
3. **ドメイン知識の明文化**: 「異常値」の定義をコードとして表現できる。

---

## 実装例：NormalizationService

PythonでValue ObjectとEntityを用いてセンサーデータを抽象化します。

```python
from dataclasses import dataclass
from typing import Final

# Value Object: 物理量を表現
@dataclass(frozen=True)
class Temperature:
    value: float
    unit: str = "Celsius"

    def __post_init__(self):
        if self.value < -273.15:
            raise ValueError("絶対零度以下の値は設定できません")

# Entity: センサーの観測データ
class SensorReading:
    def __init__(self, sensor_id: str, temperature: Temperature):
        self.sensor_id = sensor_id
        self.temperature = temperature

# Service: ドメインロジックの実行
class NormalizationService:
    def normalize(self, raw_data: dict) -> SensorReading:
        """
        外部フォーマットをドメインオブジェクトに変換・正規化する
        """
        # 単位変換ロジック（例：華氏から摂氏へ）
        temp_val = raw_data["value"]
        if raw_data["unit"] == "Fahrenheit":
            temp_val = (temp_val - 32) * 5 / 9
            
        return SensorReading(
            sensor_id=raw_data["id"],
            temperature=Temperature(value=round(temp_val, 2))
        )

# 使用例
raw = {"id": "sensor_001", "value": 98.6, "unit": "Fahrenheit"}
service = NormalizationService()
reading = service.normalize(raw)
print(f"Sensor: {reading.sensor_id}, Temp: {reading.temperature.value}°C")
```

## まとめ

DDDを導入することで、環境データのような複雑で変動しやすいデータも、ビジネスロジックから切り離して安全に管理できます。まずは「単位」や「物理的制約」をValue Objectに切り出すことから始めてみてください。
